## Overview

Deploy moto infrastructure services (moto-club, moto-keybox, PostgreSQL) to the k3d cluster as K8s workloads. This is the production-like deployment path — all components run in K8s instead of as local cargo processes.

**Scope:**
- `moto-system` namespace for infrastructure
- PostgreSQL StatefulSet for persistent data (hosts both `moto_club` and `moto_keybox` databases)
- moto-keybox and moto-club as K8s Deployments with Services
- K8s Secrets for credentials and keys (generated by Makefile target)
- RBAC: ServiceAccount + ClusterRole for moto-club
- Makefile targets for deployment lifecycle

**Out of scope:**
- Remote/cloud cluster deployment
- Helm charts (raw manifests for v1, Helm later)
- Ingress / external access (services accessed via port-forward)
- HA / multi-replica deployment
- Automated certificate management
- GitOps (ArgoCD/Flux)
- NetworkPolicy for moto-system namespace (future — garages have isolation per [garage-isolation.md](garage-isolation.md))

## Specification

### Architecture

```
k3d cluster "moto"
├── moto-system namespace
│   ├── postgres (StatefulSet)
│   │   ├── moto_club database
│   │   └── moto_keybox database
│   ├── moto-keybox (Deployment + Service)
│   │   └── auto-runs migrations on startup
│   └── moto-club (Deployment + Service)
│       └── auto-runs migrations on startup (see moto-club.md v2.3)
└── moto-garage-* namespaces (created dynamically by moto-club)
    └── garage pods
```

### Registry Names

Images are pushed to the k3d registry from the host, and pulled by pods using the in-cluster name:

| Context | Registry Address | Example |
|---------|-----------------|---------|
| Host (build/push) | `localhost:5050` | `docker push localhost:5050/moto-club:latest` |
| In-cluster (K8s pods) | `moto-registry:5000` | `image: moto-registry:5000/moto-club:latest` |

See [local-cluster.md](local-cluster.md) for registry setup details.

### Components

#### PostgreSQL

| Property | Value |
|----------|-------|
| Kind | StatefulSet |
| Image | `postgres:16-alpine` |
| Databases | `moto_club`, `moto_keybox` |
| Storage | 1Gi PVC (`postgres-data`), uses k3d default StorageClass (`local-path`) |
| User | `moto` (password from `postgres-credentials` Secret) |
| Service | `postgres.moto-system:5432` (ClusterIP) |
| Readiness | `pg_isready -U moto` probe |

**Database initialization:** `POSTGRES_DB=moto_club` creates the primary database. A ConfigMap `postgres-init` mounted at `/docker-entrypoint-initdb.d/` creates the second database:

```sql
-- postgres-init ConfigMap contents
CREATE DATABASE moto_keybox;
```

Same pattern as local dev (`scripts/init-dev-db.sql`).

#### moto-keybox

| Property | Value |
|----------|-------|
| Kind | Deployment (1 replica) |
| Image | `moto-registry:5000/moto-keybox:latest` |
| Service | `moto-keybox.moto-system:8080` (ClusterIP), port 8081 (health) |
| Resources | requests: 50m CPU / 128Mi, limits: 500m CPU / 512Mi |
| Migrations | Automatic on startup (already implemented in moto-keybox-db) |
| Health probes | Per [moto-bike.md](moto-bike.md) engine contract on port 8081 |

**Deployment-specific values:** Environment variables are defined in [keybox.md](keybox.md). The deployment sets these values:

| Variable | Value in K8s |
|----------|-------------|
| `MOTO_KEYBOX_DATABASE_URL` | From Secret `keybox-db-credentials` key `url` |
| `MOTO_KEYBOX_MASTER_KEY_FILE` | `/run/secrets/keybox/master.key` (Secret volume) |
| `MOTO_KEYBOX_SVID_SIGNING_KEY_FILE` | `/run/secrets/keybox/signing.key` (Secret volume) |
| `MOTO_KEYBOX_SERVICE_TOKEN_FILE` | `/run/secrets/keybox/service-token` (Secret volume) |

Secret `keybox-keys` mounted at `/run/secrets/keybox/` provides the key files.

#### moto-club

| Property | Value |
|----------|-------|
| Kind | Deployment (1 replica) |
| Image | `moto-registry:5000/moto-club:latest` |
| Service | `moto-club.moto-system:8080` (ClusterIP), port 8081 (health), port 9090 (metrics) |
| ServiceAccount | `moto-club` with ClusterRole (see RBAC section) |
| Resources | requests: 50m CPU / 128Mi, limits: 500m CPU / 512Mi |
| Migrations | Automatic on startup (see [moto-club.md](moto-club.md) v2.3) |
| Health probes | Per [moto-bike.md](moto-bike.md) engine contract on port 8081 |

**Deployment-specific values:** Environment variables are defined in [moto-club.md](moto-club.md). The deployment sets these values:

| Variable | Value in K8s |
|----------|-------------|
| `MOTO_CLUB_DATABASE_URL` | From Secret `club-db-credentials` key `url` |
| `MOTO_CLUB_KEYBOX_URL` | `http://moto-keybox.moto-system:8080` |
| `MOTO_CLUB_KEYBOX_HEALTH_URL` | `http://moto-keybox.moto-system:8081` |
| `MOTO_CLUB_KEYBOX_SERVICE_TOKEN_FILE` | `/run/secrets/club/service-token` (Secret volume) |
| `MOTO_CLUB_DEV_CONTAINER_IMAGE` | `moto-registry:5000/moto-garage:latest` |
| `MOTO_CLUB_DERP_SERVERS` | `[]` (empty — no DERP relay for local dev; direct UDP within k3d) |

Secret `keybox-service-token` mounted at `/run/secrets/club/` provides the service token.

### RBAC

moto-club needs a ServiceAccount with a ClusterRole granting cluster-wide access to manage garage namespaces dynamically. Required permissions:

| Resources | API Group | Verbs |
|-----------|-----------|-------|
| namespaces | core | get, list, create, delete, patch |
| pods | core | get, list, create, delete |
| services | core | get, list, create, delete |
| configmaps | core | get, list, create, delete |
| secrets | core | get, list, create, delete |
| persistentvolumeclaims | core | get, list, create, delete |
| deployments | apps | get, list, create, delete |
| networkpolicies | networking.k8s.io | get, list, create, delete |
| resourcequotas | core | get, list, create, delete |
| limitranges | core | get, list, create, delete |
| tokenreviews | authentication.k8s.io | create |

**Note:** `namespaces` needs `patch` for label updates (e.g., TTL extension via `moto garage extend`).

**Important:** The ClusterRole and ClusterRoleBinding are cluster-scoped resources — they are NOT deleted by `kubectl delete namespace`. They are cleaned up when the cluster is deleted via `dev-cluster-down`.

### Secrets

All secrets are generated by the `deploy-secrets` Makefile target. Requirements:

- Credentials and keys are generated once, stored in `.dev/k8s-secrets/` (gitignored)
- Target is idempotent — skips generation if files already exist
- Uses `moto-keybox init` for keybox keys (master.key, signing.key)
- Generates DB password and service token via random hex
- Applies all secrets to the `moto-system` namespace (creates namespace first if needed)

| Secret Name | Namespace | Contents | Used by |
|-------------|-----------|----------|---------|
| `postgres-credentials` | moto-system | `password`: generated DB password | PostgreSQL env, connection URLs |
| `keybox-keys` | moto-system | `master.key`, `signing.key`, `service-token` | moto-keybox volume mount |
| `keybox-db-credentials` | moto-system | `url`: `postgres://moto:<pw>@postgres.moto-system:5432/moto_keybox` | moto-keybox env |
| `club-db-credentials` | moto-system | `url`: `postgres://moto:<pw>@postgres.moto-system:5432/moto_club` | moto-club env |
| `keybox-service-token` | moto-system | `service-token`: same token as in `keybox-keys` | moto-club volume mount |

**Why `keybox-service-token` is separate from `keybox-keys`:** moto-club needs only the service token, not the master key or signing key. Separate secrets follow least-privilege — each service mounts only the secrets it needs.

### Container Images

Three images must be built and pushed to the local registry before deployment. Build and push targets are defined in [makefile.md](makefile.md). See [container-system.md](container-system.md) for the `mkBike` helper and Nix build pipeline.

| Image | In-cluster Reference |
|-------|---------------------|
| `moto-garage` | `moto-registry:5000/moto-garage:latest` |
| `moto-club` | `moto-registry:5000/moto-club:latest` |
| `moto-keybox` | `moto-registry:5000/moto-keybox:latest` |

### Manifest Directory

```
infra/
└── k8s/
    └── moto-system/
        ├── namespace.yaml          # Namespace with labels
        ├── postgres.yaml           # StatefulSet + Service + PVC + init ConfigMap
        ├── keybox.yaml             # Deployment + Service
        ├── club.yaml               # Deployment + Service + ServiceAccount + ClusterRole + ClusterRoleBinding
        └── kustomization.yaml      # Combines all resources
```

### Ordering Dependencies

```
namespace
  ↓
secrets (deploy-secrets)
  ↓
postgres (needs: namespace, postgres-credentials secret)
  ↓
moto-keybox (needs: postgres ready, keybox-keys + keybox-db-credentials secrets)
  ↓
moto-club (needs: postgres ready, keybox ready, club-db-credentials + keybox-service-token secrets)
  ↓
garage pods (created dynamically by moto-club)
```

K8s handles runtime ordering via readiness probes — each service won't become ready until its dependencies are healthy. The `deploy-system` target applies all manifests at once; pods start in dependency order naturally.

### Deployment Flow

**Quick path** (one command after cluster exists):

```bash
# 1. Cluster must exist
moto cluster init

# 2. Build images, generate secrets, deploy, verify (includes port-forward on localhost:18080)
make deploy

# 3. Use it
moto garage open
```

**Step-by-step** (for debugging or partial deploys):

```bash
# 1. Cluster must exist
moto cluster init

# 2. Build and push all images
make deploy-images

# 3. Generate and apply secrets
make deploy-secrets

# 4. Deploy infrastructure
make deploy-system

# 5. Wait for rollout and verify
make deploy-status

# 6. Use it
moto garage open
```

### CLI Access

`deploy-system` automatically starts a background `kubectl port-forward` from `localhost:18080` to `svc/moto-club:8080` in the `moto-system` namespace. The CLI defaults to `http://localhost:18080` for moto-club API access.

Port 18080 is chosen to avoid conflicts with common services on 80, 443, and 8080.

`dev-cluster-down` tears down the cluster (and the port-forward with it).

### Makefile Targets

| Target | Description |
|--------|-------------|
| `deploy-images` | Build and push all service images (garage, club, keybox) to local registry |
| `deploy-secrets` | Generate credentials/keys (idempotent), create K8s secrets in moto-system |
| `deploy-system` | `kubectl apply -k infra/k8s/moto-system/` |
| `deploy-status` | Wait for all Deployments to be available, then show pod/service status. Exit 0 if all healthy, exit 1 if any pod not ready. |
| `deploy` | Full deploy: `deploy-images` + `deploy-secrets` + `deploy-system` + `deploy-status` |

**Note:** `deploy-images` combines the individual `build-*` and `push-*` targets from [makefile.md](makefile.md). Without it, pods will enter `ImagePullBackOff` because the images don't exist in the registry.

### Troubleshooting

| Problem | Solution |
|---------|----------|
| Image pull fails | Verify registry from host: `curl localhost:5050/v2/_catalog` |
| Postgres pod pending | Check PVC binding: `kubectl -n moto-system get pvc` (k3d uses `local-path` StorageClass) |
| moto-keybox CrashLoopBackOff | Check logs — likely missing keys Secret or DB not ready |
| moto-club CrashLoopBackOff | Check logs — likely missing DB, keybox, or service token |
| RBAC permission denied | Check ClusterRoleBinding: `kubectl get clusterrolebinding moto-club` |
| Migrations fail | Check pod logs at startup — migration errors are logged before server starts |
| Cluster not running | `k3d cluster start moto` or `moto cluster init` to recreate |
