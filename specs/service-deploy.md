# Service Deployment

| | |
|--------|----------------------------------------------|
| Version | 0.2 |
| Status | Bare Frame |

## Overview

Deploy moto infrastructure services (moto-club, moto-keybox, PostgreSQL) to the k3d cluster as K8s workloads. This is the production-like deployment path — all components run in K8s instead of as local cargo processes.

**Scope:**
- `moto-system` namespace for infrastructure
- PostgreSQL StatefulSet for persistent data (hosts both `moto_club` and `moto_keybox` databases)
- moto-keybox and moto-club as K8s Deployments
- K8s Secrets for credentials and keys (generated by Makefile target)
- RBAC: ServiceAccount + ClusterRole for moto-club
- Makefile targets for deployment lifecycle
- Database migrations (both services auto-run on startup)

**Out of scope:**
- Remote/cloud cluster deployment
- Helm charts (raw manifests for v1, Helm later)
- Ingress / external access (services accessed via port-forward)
- HA / multi-replica deployment
- Automated certificate management
- GitOps (ArgoCD/Flux)

## Specification

### Architecture

```
k3d cluster "moto"
├── moto-system namespace
│   ├── postgres (StatefulSet)
│   │   ├── moto_club database
│   │   └── moto_keybox database
│   ├── moto-keybox (Deployment)
│   │   └── auto-runs migrations on startup
│   └── moto-club (Deployment)
│       └── auto-runs migrations on startup
└── moto-garage-* namespaces (created dynamically by moto-club)
    └── garage pods
```

### Registry Names

Images are pushed to the k3d registry from the host, and pulled by pods using the in-cluster name:

| Context | Registry Address | Example |
|---------|-----------------|---------|
| Host (build/push) | `localhost:5050` | `docker push localhost:5050/moto-club:latest` |
| In-cluster (K8s pods) | `moto-registry:5000` | `image: moto-registry:5000/moto-club:latest` |

See [local-cluster.md](local-cluster.md) for registry setup details.

### Components

#### PostgreSQL

| Property | Value |
|----------|-------|
| Kind | StatefulSet |
| Image | `postgres:16-alpine` |
| Databases | `moto_club`, `moto_keybox` |
| Storage | 1Gi PVC (`postgres-data`) |
| Credentials | Generated password, stored in K8s Secret `postgres-credentials` |
| Readiness | `pg_isready -U moto` probe |
| Init | ConfigMap `postgres-init` mounted at `/docker-entrypoint-initdb.d/` creates `moto_keybox` database |

The `POSTGRES_DB=moto_club` env var creates the primary database. The init script creates `moto_keybox` as a second database (same pattern as local dev — see `scripts/init-dev-db.sql`).

Accessible within the cluster at `postgres.moto-system.svc.cluster.local:5432`.

#### moto-keybox

| Property | Value |
|----------|-------|
| Kind | Deployment (1 replica) |
| Image | `moto-registry:5000/moto-keybox:latest` |
| Ports | 8080 (API), 8081 (health) |
| Migrations | Automatic on startup (embedded via `sqlx::migrate!()`) |
| Health probes | liveness, readiness, startup on port 8081 per [moto-bike.md](moto-bike.md) |

**Environment variables:**

| Variable | Source | Value |
|----------|--------|-------|
| `MOTO_KEYBOX_DATABASE_URL` | Secret `keybox-db-credentials` | `postgres://moto:<password>@postgres.moto-system:5432/moto_keybox` |
| `MOTO_KEYBOX_MASTER_KEY_FILE` | Secret volume mount | `/run/secrets/keybox/master.key` |
| `MOTO_KEYBOX_SVID_SIGNING_KEY_FILE` | Secret volume mount | `/run/secrets/keybox/signing.key` |
| `MOTO_KEYBOX_SERVICE_TOKEN_FILE` | Secret volume mount | `/run/secrets/keybox/service-token` |
| `MOTO_KEYBOX_BIND_ADDR` | Literal | `0.0.0.0:8080` |
| `MOTO_KEYBOX_HEALTH_BIND_ADDR` | Literal | `0.0.0.0:8081` |
| `RUST_LOG` | Literal | `moto_keybox=info` |

**Volumes:**

| Volume | Type | Mount Path |
|--------|------|------------|
| `keybox-keys` | Secret | `/run/secrets/keybox/` (master.key, signing.key, service-token) |

#### moto-club

| Property | Value |
|----------|-------|
| Kind | Deployment (1 replica) |
| Image | `moto-registry:5000/moto-club:latest` |
| Ports | 8080 (API), 8081 (health), 9090 (metrics) |
| ServiceAccount | `moto-club` with ClusterRole |
| Migrations | Automatic on startup (embedded via `sqlx::migrate!()`, same pattern as keybox) |
| Health probes | liveness, readiness, startup on port 8081 per [moto-bike.md](moto-bike.md) |

**Environment variables:**

| Variable | Source | Value |
|----------|--------|-------|
| `MOTO_CLUB_DATABASE_URL` | Secret `club-db-credentials` | `postgres://moto:<password>@postgres.moto-system:5432/moto_club` |
| `MOTO_CLUB_KEYBOX_URL` | Literal | `http://moto-keybox.moto-system:8080` |
| `MOTO_CLUB_KEYBOX_HEALTH_URL` | Literal | `http://moto-keybox.moto-system:8081` |
| `MOTO_CLUB_KEYBOX_SERVICE_TOKEN_FILE` | Secret volume mount | `/run/secrets/club/service-token` |
| `MOTO_CLUB_DEV_CONTAINER_IMAGE` | Literal | `moto-registry:5000/moto-garage:latest` |
| `MOTO_CLUB_BIND_ADDR` | Literal | `0.0.0.0:8080` |
| `RUST_LOG` | Literal | `moto_club=info` |

**Volumes:**

| Volume | Type | Mount Path |
|--------|------|------------|
| `club-service-token` | Secret (`keybox-service-token`) | `/run/secrets/club/` (service-token) |

**Migration prerequisite:** moto-club-db must embed migrations via `sqlx::migrate!()` and expose `pub async fn run_migrations(pool)` — same pattern as moto-keybox-db. The moto-club binary calls this on startup before serving requests. This keeps the bike image small (no sqlx-cli needed).

### Health Probes

All services use the [moto-bike.md](moto-bike.md) engine contract for health:

```yaml
livenessProbe:
  httpGet:
    path: /health/live
    port: 8081
  initialDelaySeconds: 5
  periodSeconds: 10
readinessProbe:
  httpGet:
    path: /health/ready
    port: 8081
  initialDelaySeconds: 5
  periodSeconds: 5
startupProbe:
  httpGet:
    path: /health/startup
    port: 8081
  failureThreshold: 30
  periodSeconds: 2
```

### RBAC

moto-club needs a ServiceAccount with a ClusterRole granting cluster-wide access to manage garage namespaces dynamically. Required permissions:

| Resources | API Group | Verbs |
|-----------|-----------|-------|
| namespaces | core | get, list, create, delete, patch |
| pods | core | get, list, create, delete |
| services | core | get, list, create, delete |
| configmaps | core | get, list, create, delete |
| secrets | core | get, list, create, delete |
| persistentvolumeclaims | core | get, list, create, delete |
| deployments | apps | get, list, create, delete |
| networkpolicies | networking.k8s.io | get, list, create, delete |
| resourcequotas | core | get, list, create, delete |
| limitranges | core | get, list, create, delete |
| tokenreviews | authentication.k8s.io | create |

**Note:** `namespaces` needs `patch` for label updates (e.g., TTL extension via `moto garage extend`).

### Secrets

All secrets are generated by the `deploy-secrets` Makefile target. Keys and tokens are generated once and stored in `.dev/k8s-secrets/` (gitignored), then applied as K8s Secrets.

| Secret Name | Namespace | Contents | Used by |
|-------------|-----------|----------|---------|
| `postgres-credentials` | moto-system | `password`: generated DB password | PostgreSQL env, connection URLs |
| `keybox-keys` | moto-system | `master.key`, `signing.key`, `service-token` | moto-keybox volume mount |
| `keybox-db-credentials` | moto-system | `url`: `postgres://moto:<pw>@postgres.moto-system:5432/moto_keybox` | moto-keybox env |
| `club-db-credentials` | moto-system | `url`: `postgres://moto:<pw>@postgres.moto-system:5432/moto_club` | moto-club env |
| `keybox-service-token` | moto-system | `service-token`: same token as in `keybox-keys` | moto-club volume mount |

**`deploy-secrets` implementation:**

```bash
# Generate password and keys (idempotent — skip if .dev/k8s-secrets/ exists)
mkdir -p .dev/k8s-secrets
[ -f .dev/k8s-secrets/db-password ] || openssl rand -hex 16 > .dev/k8s-secrets/db-password
[ -f .dev/k8s-secrets/master.key ] || cargo run --bin moto-keybox-cli -- init --output-dir .dev/k8s-secrets
[ -f .dev/k8s-secrets/service-token ] || openssl rand -hex 32 > .dev/k8s-secrets/service-token

# Create namespace first
kubectl apply -f infra/k8s/moto-system/namespace.yaml

# Apply secrets
kubectl -n moto-system create secret generic postgres-credentials \
  --from-file=password=.dev/k8s-secrets/db-password --dry-run=client -o yaml | kubectl apply -f -
# ... (similar for other secrets)
```

### Container Images

Three images must be built and pushed to the local registry (`localhost:5050`) before deployment:

| Image | Nix Package | Push Command | In-cluster Reference |
|-------|-------------|--------------|---------------------|
| `moto-garage` | `moto-garage` | `make push-garage` | `moto-registry:5000/moto-garage:latest` |
| `moto-club` | `moto-club-image` | `make push-club` | `moto-registry:5000/moto-club:latest` |
| `moto-keybox` | `moto-keybox-image` | `make push-keybox` | `moto-registry:5000/moto-keybox:latest` |

Service images use the `mkBike` helper. See [container-system.md](container-system.md).

### Manifest Directory

```
infra/
└── k8s/
    └── moto-system/
        ├── namespace.yaml          # Namespace with labels
        ├── postgres.yaml           # StatefulSet + Service + PVC + init ConfigMap
        ├── keybox.yaml             # Deployment + Service
        ├── club.yaml               # Deployment + Service + ServiceAccount + ClusterRole + ClusterRoleBinding
        └── kustomization.yaml      # Combines all resources
```

### Ordering Dependencies

```
namespace
  ↓
secrets (deploy-secrets)
  ↓
postgres (needs: namespace, postgres-credentials secret)
  ↓
moto-keybox (needs: postgres ready, keybox-keys + keybox-db-credentials secrets)
  ↓
moto-club (needs: postgres ready, keybox ready, club-db-credentials + keybox-service-token secrets)
  ↓
garage pods (created dynamically by moto-club)
```

K8s handles runtime ordering via readiness probes — each service won't become ready until its dependencies are healthy. The `deploy-system` target applies all manifests at once; pods start in dependency order naturally.

### Deployment Flow

```bash
# 1. Cluster must exist
moto cluster init

# 2. Build and push all images
make build-garage push-garage
make build-club push-club
make build-keybox push-keybox

# 3. Generate and apply secrets
make deploy-secrets

# 4. Deploy infrastructure
make deploy-system

# 5. Wait for rollout
make deploy-status

# 6. Port-forward for CLI access
kubectl -n moto-system port-forward svc/moto-club 8080:8080 &

# 7. Use it
moto garage open
```

### CLI Access

The CLI needs to reach moto-club. Two options:

| Option | Pros | Cons |
|--------|------|------|
| Port-forward | Simple, no config changes | Background process, can drop |
| NodePort service | Stable, no background process | Requires additional Service resource |

V1 uses port-forward. NodePort can be added later if needed.

### Makefile Targets

| Target | Description |
|--------|-------------|
| `deploy-secrets` | Generate credentials/keys (idempotent), create K8s secrets in moto-system |
| `deploy-system` | `kubectl apply -k infra/k8s/moto-system/` |
| `deploy-status` | Show pod/service status in moto-system namespace |
| `undeploy-system` | `kubectl delete namespace moto-system` (cascades all resources) |

**Note:** `build-club`, `push-club`, `build-keybox`, `push-keybox` already exist in the Makefile (per [makefile.md](makefile.md)).

### Troubleshooting

| Problem | Solution |
|---------|----------|
| Image pull fails | Verify registry from host: `curl localhost:5050/v2/_catalog` |
| Postgres pod pending | Check PVC binding: `kubectl -n moto-system get pvc` |
| moto-keybox CrashLoopBackOff | Check logs — likely missing keys Secret or DB not ready |
| moto-club CrashLoopBackOff | Check logs — likely missing DB, keybox, or service token |
| RBAC permission denied | Check ClusterRoleBinding: `kubectl get clusterrolebinding moto-club` |
| Migrations fail | Check pod logs at startup: `kubectl -n moto-system logs <pod>` — migration errors are logged before server starts |

## Deferred

- Helm chart packaging
- Ingress controller for external access
- TLS termination
- Multi-replica deployments
- Automated database backups
- Prometheus / Grafana monitoring stack
